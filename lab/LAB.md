# Лабораторная: добавь микросервис в сагу

## Цель

Добавить в существующую сагу «оформление заказа» новый микросервис **antifraud**
(антифрод-проверка) — и сделать это в **обоих** вариантах: оркестрация и
хореография. По ходу — увидеть на практике, чем отличается внесение изменений в
этих двух подходах.

## Чему научишься

- Добавлять новый сервис как самостоятельную единицу (свой код, БД, Dockerfile, контракт).
- Встраивать новый шаг в распределённую транзакцию (Saga) и его компенсацию/откат.
- На своём опыте почувствовать ключевую разницу: где живёт логика процесса и
  сколько мест приходится трогать при изменении в оркестрации vs хореографии.

## Что уже есть

Базовая сага (см. корневой `README.md`):

```
Заказ → Резерв на складе → Списание оплаты → Доставка
```

- **orchestration/** — центральный `orchestrator` синхронно по шагам дёргает сервисы.
- **choreography/** — сервисы общаются событиями через RabbitMQ, без дирижёра.

Стартовая точка лабы — этот код **без** сервиса antifraud.

## Задание

Вставить новый шаг **antifraud** между резервом склада и оплатой:

```
Заказ → Склад(резерв) → [antifraud] → Оплата → Доставка
```

Идея: перед тем как списывать деньги, проверяем заказ на мошенничество. Если
проверка не прошла — заказ отклоняется, а уже сделанный резерв склада
откатывается (компенсация).

### Контракт сервиса antifraud (обязателен — его проверяет автотест)

- Решение: **reject**, если `amount > FRAUD_LIMIT` (переменная окружения,
  по умолчанию `3000`); иначе **pass**.
- Своя БД (как у других сервисов — локальный SQLite через SQLAlchemy): таблица
  `checks(order_id, decision)`, где `decision ∈ {passed, rejected}`.
- HTTP-эндпоинт для чтения решения (нужен и в оркестрации, и в хореографии):
  `GET /antifraud/{order_id}` → `{"order_id": ..., "decision": "passed"|"rejected"|"unknown"}`.
- `GET /health` → `{"ok": true}`.
- Порог берётся из `FRAUD_LIMIT`, никаких «магических чисел» в коде.

### Часть 1. Оркестрация (`orchestration/`)

1. Создай сервис `orchestration/antifraud/` (свой `main.py`, `db.py`,
   `Dockerfile`, `requirements.txt`). В оркестрации antifraud — обычный HTTP-сервис:
   `POST /check {order_id, amount}` → `200` если прошёл, `403` если отклонён
   (и записывает решение в свою БД). Плюс `GET /antifraud/{id}` и `/health`.
2. В `orchestrator/main.py` вставь вызов antifraud **между** `reserve` и `charge`.
   При отказе — запусти компенсацию уже выполненных шагов (release склада) и верни
   `{"status": "FAILED", "failed_at": "antifraud"}`.
3. В `orchestration/docker-compose.yml` добавь сервис `antifraud` (порт `8010:8000`,
   `FRAUD_LIMIT=3000`), пропиши `orchestrator` зависимость и `ANTIFRAUD_URL`.

### Часть 2. Хореография (`choreography/`)

1. Создай сервис `choreography/antifraud/`. Он **слушает** `inventory.reserved`
   и **публикует**:
   - `antifraud.passed` (с тем же payload) — если проверка пройдена;
   - `antifraud.rejected` + `order.failed` (`failed_at: "antifraud"`) — если отклонена.
   Также подними у него небольшой HTTP-API (`GET /antifraud/{id}`, `/health`) —
   так же, как сервис `order` совмещает API и слушателя.
2. **Перепаяй цепочку:**
   - `payment` теперь слушает `antifraud.passed` (а не `inventory.reserved`);
   - `inventory` должен делать компенсацию (release) ещё и по `antifraud.rejected`.
3. В `choreography/docker-compose.yml` добавь сервис `antifraud`
   (`RABBIT_URL`, `SAGA_EXCHANGE`, `FRAUD_LIMIT`, порт `8010:8000`).

## Как проверить себя (автотест)

В репозитории есть автотест `lab/autotest/` — он сам гоняет сагу по HTTP и
проверяет поведение (чёрный ящик, один и тот же тест для обоих вариантов).
Команды запуска — в `lab/autotest/README.md`. Что именно проверяется:

1. **Доступность** — `order`/`orchestrator` и `antifraud` отвечают на `/health`.
2. **Обычный заказ** — проходит антифрод и завершается: статус `COMPLETED`,
   решение антифрода `passed`.
3. **Подозрительный заказ** (`amount > FRAUD_LIMIT`) — отклоняется: статус
   `FAILED`, решение `rejected`, а где есть `failed_at` — он равен `antifraud`.
4. **Компенсация склада** — что отклонённый антифродом заказ откатывает уже
   сделанный резерв. Проверяется так: первый «фрод»-заказ занимает **весь** склад
   и отклоняется антифродом; затем легитимный заказ на тот же объём должен пройти
   (`COMPLETED`). Это возможно только если зарезервированный товар вернулся на
   склад — то есть компенсация сработала. Если в хореографии забыть про
   release по `antifraud.rejected` (или в оркестрации — про откат при отказе
   `/check`), склад останется пустым и этот тест упадёт.

> Проверку компенсации гоняй на **свежеподнятом** стеке: тест исходит из того,
> что склад стартует с `INITIAL_STOCK` (по умолчанию `5`). Если меняешь это
> значение в `docker-compose.yml`, передай такое же тесту переменной
> `INITIAL_STOCK`.
