# Хореография vs Оркестрация (микросервисы)

Мини-проект: Хореография vs Оркестрация. Один и тот же сценарий —
**оформление заказа** — реализован двумя способами, чтобы можно было поднять
сервисы локально и руками увидеть разницу.

## Микросервисы

**Микросервисная архитектура** — это когда приложение разбито на набор
маленьких независимых сервисов, каждый из которых отвечает за свою область
(склад, оплата, доставка) и общается с остальными по сети (HTTP или события).
Противоположность — монолит, где всё это живёт в одном процессе и одной БД.

Что это даёт: сервисы можно разрабатывать, деплоить и масштабировать
независимо, разными командами и даже на разных языках. Цена — распределённость:
сеть ненадёжна, общей транзакции на всех нет, и согласованность приходится
обеспечивать вручную (об этом — раздел про Saga ниже).

### У каждого сервиса — своя база данных

Ключевое правило микросервисов: **database per service**. Каждый сервис владеет
своими данными и имеет собственную БД, к которой больше никто не ходит напрямую.
Чужие данные можно получить только через API или события этого сервиса.

Почему так:

- **Слабая связанность.** Сервис может менять свою схему БД, не ломая других —
  никто не завязан на его таблицы.
- **Независимый деплой и масштабирование.** БД склада можно тюнить/масштабировать
  отдельно от БД оплаты.
- **Изоляция отказов.** Проблемы в одной БД не кладут остальные сервисы.
- **Polyglot persistence.** Каждому — подходящее хранилище (где-то PostgreSQL,
  где-то Redis или MongoDB).

Чего так делать нельзя: одна общая БД на всех (shared database) — это
«распределённый монолит», главный антипаттерн.

В этом проекте каждый сервис с состоянием держит данные в **своей локальной
БД — отдельном SQLite-файле** (`<сервис>.db`, через SQLAlchemy). У склада,
оплаты и сервиса заказов — свои независимые базы; никто не лезет в чужую.

Почему SQLite, а не Postgres? Принцип «БД на сервис» — про то, что у каждого
сервиса *своя отдельная* база, а не про конкретную СУБД. Для учебного демо
важно показать именно архитектуру, а не возиться с инфраструктурой, поэтому
берём встроенный SQLite: ничего поднимать не надо, у каждого сервиса свой файл.
В проде SQLite меняется на свой PostgreSQL/MySQL на сервис — благодаря
SQLAlchemy это правка одной строки `DATABASE_URL`, код не трогаем.

Сервисы без собственных данных своей БД не имеют — и это нормально.

## Сценарий

Классическая распределённая транзакция (паттерн **Saga**):

```
Заказ  →  Резерв на складе  →  Списание оплаты  →  Планирование доставки
```

У шагов есть компенсации (обратные действия) на случай сбоя:
оплата → возврат денег, склад → возврат товара.

## Два подхода в одном предложении

| | Оркестрация | Хореография |
|---|---|---|
| Кто управляет | один центральный сервис-«дирижёр» | никто; каждый сервис сам реагирует на события |
| Связь | команды (обычно синхронные вызовы) | события (асинхронные, через брокер) |
| Где живёт логика процесса | в одном месте (оркестратор) | размазана по сервисам |
| Метафора | дирижёр оркестра | танцоры, знающие свои движения |

### Оркестрация

```
                 ┌──────────────┐
   POST /orders  │ ORCHESTRATOR │
  ──────────────▶│  (дирижёр)   │
                 └──────┬───────┘
        1. reserve      │   2. charge        3. schedule
        ┌───────────────┼────────────────┬───────────────┐
        ▼               ▼                 ▼               
   ┌─────────┐     ┌─────────┐      ┌──────────┐
   │inventory│     │ payment │      │ shipping │
   └─────────┘     └─────────┘      └──────────┘
   Если шаг падает — оркестратор сам вызывает refund/release.
```

Поток управления явный и читается сверху вниз в одном файле:
`orchestration/orchestrator/main.py`.

### Хореография

```
   POST /orders
  ──────────────▶ ┌───────┐  order.created
                  │ order │ ───────────────┐
                  └───────┘                ▼
                                      ┌───────────┐ inventory.reserved
                                      │ inventory │ ───────────────┐
                                      └───────────┘                ▼
                                                            ┌─────────┐ payment.completed
                                                            │ payment │ ──────────────┐
                                                            └─────────┘               ▼
                                                                                ┌──────────┐
                                                                                │ shipping │
                                                                                └────┬─────┘
                                                            order.completed ◀───────┘
            Все события идут через брокер RabbitMQ (exchange `saga`).
            payment.failed → inventory сам делает компенсацию.
```

Центрального файла с логикой нет — цепочку нужно собирать «в голове»
по подпискам каждого сервиса.

## Что делает каждый сервис

Каждый сервис с состоянием держит данные в **своей локальной БД** (SQLite-файл,
через SQLAlchemy; модуль `db.py` в сервисе). Доступ — только через API/события
этого сервиса, чужие таблицы напрямую не читаются.

### Оркестрация (`orchestration/`, синхронные HTTP)

- **orchestrator** — принимает `POST /orders` и по шагам вызывает остальных:
  `reserve` на складе → `charge` в оплате → `schedule` в доставке. Если шаг
  падает, сам вызывает компенсации уже выполненных шагов в обратном порядке
  (`refund`, `release`). Координатор без бизнес-данных — **своей БД не имеет**.
- **inventory** — `POST /reserve` резервирует товар, `POST /release` —
  компенсация. БД (SQLite): таблицы `stock` и `reservations`. При нехватке — `409`.
- **payment** — `POST /charge` списывает сумму (лимит из `PAYMENT_LIMIT`; выше —
  `402`), `POST /refund` — компенсация. БД (SQLite): таблица `charges`.
- **shipping** — `POST /schedule` планирует доставку. Последний шаг, «отправить
  и забыть»: ничего не хранит — **своей БД не имеет**.

### Хореография (`choreography/`, события через RabbitMQ)

Все события идут через topic-exchange `saga`. Каждый сервис сам подписывается
на нужные ему события и публикует свои; центрального координатора нет.
`broker.py` — небольшой хелпер для RabbitMQ (подключение с ретраями, publish,
consume).

| Сервис | Слушает | Делает | Публикует |
|--------|---------|--------|-----------|
| **order** | `order.completed`, `order.failed` | принимает `POST /orders`, хранит статусы; `GET /orders/{id}` | `order.created` |
| **inventory** | `order.created`, `payment.failed` | резервирует товар; по `payment.failed` — компенсация | `inventory.reserved` / `inventory.failed` (+ `order.failed`) |
| **payment** | `inventory.reserved` | списывает деньги (лимит 1000) | `payment.completed` / `payment.failed` (+ `order.failed`) |
| **shipping** | `payment.completed` | планирует доставку | `order.completed` |

Ответ на `POST /orders` приходит сразу со статусом `PENDING`; финальный статус
(`COMPLETED` / `FAILED`) сервис заказов узнаёт позже из событий — смотри
`GET /orders/{id}`.

## Настройки

В коде нет «магических чисел» — все параметры читаются из переменных окружения
(в коде только дефолты) и задаются в `docker-compose.yml`. Меняешь поведение
демо, не трогая код.

## Как запустить

Нужен **Docker** и **Docker Compose**. Сервисы написаны на Python (FastAPI),
у каждого сервиса с состоянием — своя локальная БД (SQLite, внутри сервиса), а в
варианте с хореографией ещё поднимается брокер **RabbitMQ**. `docker compose up`
поднимет всё сам.

### Вариант 1. Оркестрация

```bash
cd orchestration
docker-compose up --build
```

В другом терминале отправляем заказ:

```bash
# Успешный заказ
curl -X POST localhost:8000/orders \
  -H 'content-type: application/json' \
  -d '{"order_id":"A1","quantity":1,"amount":100}'
# -> {"order_id":"A1","status":"COMPLETED"}
```

Смотрим логи `docker-compose` — видно, как оркестратор по очереди дёргает
склад, оплату и доставку.

**Сломаем оплату** (сумма больше лимита 1000 → банк отклоняет, запускается
компенсация — возврат товара на склад):

```bash
curl -X POST localhost:8000/orders \
  -H 'content-type: application/json' \
  -d '{"order_id":"A2","quantity":1,"amount":5000}'
# -> {"order_id":"A2","status":"FAILED","failed_at":"payment"}
```

В логах появится `КОМПЕНСАЦИЯ: возвращено ... ед.`

### Вариант 2. Хореография

```bash
cd choreography
docker compose up --build
```

Отправляем заказ так же на порт 8000 (сервис `order`):

```bash
curl -X POST localhost:8000/orders \
  -H 'content-type: application/json' \
  -d '{"order_id":"B1","quantity":1,"amount":100}'
# -> {"order_id":"B1","status":"PENDING"}   (ответ приходит сразу, асинхронно)

# Узнать финальный статус:
curl localhost:8000/orders/B1
# -> {"status":"COMPLETED", ...}
```

В логах видно, как событие проходит по цепочке `order → inventory → payment
→ shipping` без какого-либо координатора. Веб-консоль брокера:
<http://localhost:15672> (логин/пароль `guest`/`guest`) — там видно очереди и
exchange `saga`.

**Сломаем оплату** (компенсация склада запускается по событию `payment.failed`):

```bash
curl -X POST localhost:8000/orders \
  -H 'content-type: application/json' \
  -d '{"order_id":"B2","quantity":1,"amount":5000}'
curl localhost:8000/orders/B2     # -> FAILED
```

Остановить: `Ctrl+C`, затем `docker compose down`.

## Что важно заметить, запустив оба варианта

1. **Где искать логику.** В оркестрации весь сценарий — в одном сервисе. В
   хореографии его нет нигде целиком; чтобы понять процесс, надо собрать
   подписки всех сервисов.
2. **Связанность.** Оркестратор знает про все сервисы (адреса, контракты). В
   хореографии сервисы знают только про события, но не друг про друга.
3. **Синхронность.** В оркестрации клиент ждёт весь процесс. В хореографии
   ответ `PENDING` приходит мгновенно, результат — позже.
4. **Добавление шага.** В оркестрации — правим оркестратор. В хореографии —
   добавляем новый сервис-подписчик, никого не трогая (но рискуем получить
   «неявные» цепочки, которые трудно отследить).

## Когда что выбирать

- **Оркестрация** — когда процесс сложный, важны явный контроль, наблюдаемость,
  понятные компенсации. Инструменты: Camunda/Zeebe, Temporal, AWS Step Functions.
- **Хореография** — когда нужна слабая связанность, независимые команды и
  масштабирование, а сценарии простые/расширяемые. Инструменты: брокеры
  сообщений (RabbitMQ, Kafka), event-driven архитектура.

На практике часто комбинируют: внутри домена — хореография на событиях, а
сложные многошаговые саги оборачивают в оркестратор.

## Структура репозитория

Каждый сервис — **отдельный самостоятельный проект** со своим `Dockerfile` и
`requirements.txt` (как и положено микросервисам). `docker compose` собирает
каждый из своей папки.

`db.py` — доступ к локальной БД сервиса (SQLAlchemy, SQLite-файл). Есть только у
сервисов с состоянием; `orchestrator` и `shipping` его не имеют.


> `db.py` (своя БД) у каждого сервиса свой — сервисы независимы и не делят общую
> базу. Хелпер `broker.py` (RabbitMQ) вынесен в общий модуль `common/`: это
> чисто технический код доступа к брокеру, дублировать его по сервисам смысла
> нет (в отличие от данных и бизнес-логики).

> Бизнес-логика упрощена (без ретраев, идемпотентности и блокировок строк), БД —
> локальный SQLite: цель показать разницу подходов и принцип «БД на сервис», а не
> продакшен. В бою SQLite меняется на свой Postgres/MySQL на сервис.

## Лабораторная работа

В папке `lab/` лежит лабораторная: **добавить в сагу новый микросервис**
`antifraud` (антифрод-проверка) — и сделать это в обоих вариантах, оркестрации и
хореографии. Так на практике видно ключевую разницу: в оркестрации новый шаг —
это правка одного файла, а в хореографии приходится перепаять цепочку событий в
нескольких сервисах.

- Задание целиком — в `lab/LAB.md` (цель, контракт сервиса, шаги, критерии, рубрика).
- Лаба **самопроверяемая**: автотест `lab/autotest/` сам гоняет сагу по HTTP и
  проверяет поведение в обоих вариантах. Как запускать — в `lab/autotest/README.md`.
